# North — Application Architecture

Status: target architecture
Last updated: 2026-07-14

## Decision

North will remain an installable React/TypeScript PWA and evolve into an
offline-first client backed by a versioned HTTP API and PostgreSQL.

```text
React PWA
  ↓ repository interfaces
local durable store
  ↓ outbox / sync engine
versioned North API
  ↓ authenticated user boundary
PostgreSQL
```

The current local-storage prototype is a product proof, not the final storage
architecture. Product components must not learn database or authentication
provider details.

## Responsibilities

### PWA client

- phone-first interaction
- optimistic local writes
- workout operation without a network connection
- local outbox of unsynchronised changes
- explicit sync state and recoverable errors
- versioned backup export and restore

### Repository layer

UI code reads and writes domain objects through repositories:

- SessionRepository
- PlanRepository
- ActivityRepository
- CheckInRepository
- ReviewRepository
- ConversationRepository
- ProfileRepository

The first adapter wraps current browser storage. The durable adapter should use
IndexedDB. A sync decorator sends committed local changes to the API.

### North API

- validates every payload
- derives the authenticated owner from the session, never request JSON
- applies idempotent mutations
- returns server revisions and deletion tombstones
- performs privileged Nova/tool work away from the browser
- never exposes model or service credentials to the PWA

### PostgreSQL

- UUID primary keys generated by the client or server
- `owner_user_id` on every private aggregate
- UTC timestamps plus explicit activity/workout dates
- soft-delete timestamps for synchronised user records
- JSON only for bounded flexible metadata, not core relationships

## Authentication boundary

Authentication must support standards-based sessions and map one external
identity to one North profile. The provider has not been chosen yet.

Requirements before selection:

- secure browser sessions without locally stored long-lived bearer tokens
- account recovery
- email change and deletion flows
- server-verifiable identity
- support for future Apple and Google sign-in without changing North IDs

North uses its own UUID as the durable owner ID. Provider subjects are mapping
data, not primary keys throughout the product.

## Offline and synchronisation rules

1. Workout set logging always succeeds locally first.
2. Every mutation has a UUID, client timestamp, entity revision, and device ID.
3. The API treats repeated mutation IDs as the same request.
4. Append-only workout sets and activities merge by UUID.
5. Notes and scalar preferences use latest accepted revision with conflict copy
   preservation; North must never silently discard user text.
6. Plan-day conflicts are returned for explicit resolution if both devices
   changed the same day from the same base revision.
7. Deletes synchronise as tombstones until every active device has observed
   them.
8. An active workout remains editable offline and must never be replaced by an
   older server snapshot.

## Privacy and safety

- collect only information needed to guide the traveller
- keep health-adjacent records private by default
- encrypt transport and managed storage
- log operational identifiers, never workout notes or conversations
- make export and deletion understandable and available
- keep learned observations traceable to evidence
- keep Nova recommendations separate from medical diagnosis

## Migration path from the prototype

1. Stabilise the gym-floor loop through real use.
2. Move browser persistence behind repository interfaces.
3. Replace localStorage with IndexedDB and migrate existing version-1 data.
4. Stand up PostgreSQL and apply `db/migrations/0001_initial.sql`.
5. Select authentication and implement identity mapping.
6. Add the versioned API and outbox synchronisation.
7. Import the user’s versioned North backup into their first real account.

## Explicitly unresolved

- hosting and authentication provider
- regional data residency choice
- subscription and billing architecture
- model provider for Nova
- retention periods for deleted data and server backups

These decisions require deployment, privacy, and operating-cost context. They
must not be smuggled into UI code as accidental defaults.
